# 高频电商价格指数系统——架构与初步设计报告

## 一、项目概述

本项目旨在构建一套基于电商平台每日商品价格与销量数据的高频价格指数计算系统。系统需处理每日千万级商品明细数据，通过列式数据库实现秒级聚合计算，输出日度价格指数趋势，为宏观经济监测提供更灵敏的量化依据。

核心业务目标：

- 每日自动化采集/读取电商商品报价与销量数据
- 计算各层级（SKU、类目、大盘）的价格指数
- 支持日度指数趋势图的可视化输出

## 二、架构设计演进（V1 → V2）

### 2.1 V1：无约束下的“理想化”架构

在未施加任何工程约束时，AI倾向于推荐一套完整的企业级大数据架构：**Kafka + Flink/Spark Streaming + HDFS + Hive + ClickHouse + Airflow**。这套方案在技术层面无可指摘，能够覆盖实时流处理、离线批处理、数据湖存储等全部场景。

**为何最终拒绝？** 本团队人力有限、无专职运维，引入 Kafka + Flink + HDFS 意味着至少需要维护5个以上分布式组件，环境搭建与日常运维成本远超项目承受能力。这套“重架构”虽然完美，但不符合我们的工程约束。

### 2.2 V2：加入约束后的轻量化架构

向AI施加“海量数据、高聚合需求、小团队无专职运维”三个约束后，方案被大幅精简，最终保留的核心组件为：

| 组件     | 选型                    | 职责            |
| ------ | --------------------- | ------------- |
| 对象存储   | 阿里云 OSS               | 海量原始数据低成本归档   |
| 分析型数据库 | ClickHouse            | 列式存储 + 秒级聚合查询 |
| 计算与调度  | Python + 定时任务         | 轻量级ETL与流程编排   |
| 可视化    | Matplotlib / Quick BI | 日度指数趋势图输出     |
| CI/CD  | GitHub Actions        | 自动化测试与部署      |

**精简逻辑**：用对象存储替代HDFS（运维成本趋近于零），用ClickHouse内置计算能力替代Spark（避免维护计算集群），用Python脚本 + cron替代Airflow（无需部署调度器）。

## 三、核心数据流水线（E-L-T）

系统遵循经典的**提取-加载-转换**（ELT）流水线设计：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        1. 数据提取 (Extract)                          │
│              定时任务触发，读取外部CSV/API获取原始数据                   │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                   2. 加载到数据湖 (Load to Data Lake)                  │
│              原始数据完整备份至阿里云OSS（低成本归档）                   │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                  3. 加载到OLAP引擎 (Load to OLAP)                     │
│          通过S3表引擎或表函数将OSS数据导入ClickHouse        │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                   4. 引擎内计算 (Transform & Compute)                  │
│           在ClickHouse中执行Group By聚合 + 费雪指数计算                │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                        5. 结果分发 (Serve)                            │
│              写入结果表 → 可视化大屏 / API接口输出                      │
└─────────────────────────────────────────────────────────────────────────┘
```

**关键设计原则**：原始数据必须先落OSS再导入ClickHouse。这确保了无论下游计算发生何种错误，原始底稿永远可追溯、可重算。

## 五、Schema 设计（ClickHouse）

### 5.1 分类维度表（category）

```sql
CREATE TABLE category (
    category_id   UInt64,             -- 分类ID（如 1101000000）
    category      String,             -- 分类名称
    hierarchy     UInt8,              -- 层级（1=一级, 2=二级）
    weight        Float32,            -- 权重
    price         Nullable(Float32),  -- 参考价格（一级分类可为NULL）
    parent        Nullable(UInt64),   -- 父分类ID（一级分类为NULL）
    load_time     DateTime DEFAULT now()
) ENGINE = MergeTree()
ORDER BY (hierarchy, category_id)
SETTINGS index_granularity = 8192;
```

### 5.2 商品全量表（product_full）

```sql
CREATE TABLE product_full (
    product_id    UInt64,             -- 商品唯一标识（如 945831417949）
    category_id   UInt64,             -- 所属分类ID
    name          String,             -- 商品名称
    weight        Float32,            -- 权重
    price         Float32,            -- 当前价格
    change_count  UInt32,             -- 变更次数
    dt            Date,               -- 快照日期（分区字段）
    load_time     DateTime DEFAULT now()
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(dt)
ORDER BY (dt, category_id, product_id)
SETTINGS index_granularity = 8192;
```

**设计理由**：

- **分区键 `toYYYYMM(dt)`**：按月切割数据，查询时仅扫描目标月份。
- **排序键 `(dt, category_id, product_id)`**：查询模式以日期+分类筛选为主，此顺序可最大化索引命中率。
- **ENGINE = MergeTree()**：ClickHouse最通用的表引擎，支持主键索引和数据分区。

### 5.3 每日价格快照表（product_price）

```sql
CREATE TABLE product_price (
    product_id    UInt64,             -- 商品唯一标识
    category_id   UInt64,             -- 所属分类ID
    name          String,             -- 商品名称
    price         Float32,            -- 当日价格
    change_date   Date,               -- 价格日期
    load_time     DateTime DEFAULT now()
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(change_date)
ORDER BY (change_date, category_id, product_id)
SETTINGS index_granularity = 8192;
```

### 5.4 指数结果表（index_result）

```sql
CREATE TABLE index_result (
    date          Date,               -- 计算日期
    index_level   String,             -- 'total' / 'category'
    category_id   UInt64,             -- 分类ID（总指数时为 0）
    category_name String,             -- 分类名称（总指数时为 'ALL'）
    index_type    String,             -- 'laspeyres' / 'fisher'
    index_value   Float64,            -- 指数值
    base_date     Date,               -- 基期日期
    compute_time  DateTime DEFAULT now()
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(date)
ORDER BY (date, index_level, category_id);
```

## 六、核心算法选型

### 6.1 底层SKU处理：几何平均（Jevons Index）

同一SKU一天内可能存在多个报价，采用几何平均可有效抹平极端异常值的影响，避免个别商家的异常定价扭曲整体指数。

### 6.2 聚合层算法

- **初级方案——拉氏指数（Laspeyres Index）** ：以基期销量为权重，实现简单，但会高估物价变动（忽略消费者替代效应）。
- **高阶方案——费雪理想指数（Fisher Index）** ：结合基期销量（拉氏）与当期销量（派氏）求几何平均，更真实地反映市场动态。由于电商数据可提供每日实时销量，本项目优先采用费雪指数。

## 七、测试策略

### 7.1 单元测试（计算逻辑）

手工构造极小规模边界数据集（如3个商品，其中1个销量骤降为0），用 `pytest` 断言费雪指数公式在极端场景下的数学正确性。

### 7.2 集成测试（数据流转）

通过 `testcontainers` 启动轻量级 ClickHouse Docker 容器，写入百条模拟数据，验证“清洗 → 上传OSS → 导入ClickHouse → 聚合计算 → 写入结果表”全流程。

### 7.3 数据质量监控（运行时防御）

在调度代码中埋入监控点：计算当日总指数后与昨日比对，若单日涨跌幅超过50%则自动告警并熔断——说明源数据极可能被污染。

## 八、CI/CD 自动化（GitHub Actions）

配置 GitHub Actions 实现代码提交后自动触发：

1. 代码格式校验与静态检查
2. 单元测试（小规模CSV验证SQL正确性）
3. 集成测试（Docker启动ClickHouse进行全流程验证）
4. （可选）自动部署至生产环境

## 十、总结

本架构的核心思想是 **“存算分离、轻量运维”** ：用OSS承载海量原始数据（低成本存储），用ClickHouse承载计算（高性能OLAP），用Python脚本串联流程（低耦合）。从V1的“重架构”到V2的“轻量化”，本质上是在**功能完备性**与**运维可行性**之间做出的务实取舍——这正是真实工程世界中架构师的核心工作。
